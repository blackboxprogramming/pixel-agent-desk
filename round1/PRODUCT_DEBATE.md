#### ⚠️ 심각한 문제점 발견

**문제 1: PRD와 현실의 괴리 (가치 제안 불일치)**

**PRD 주장:**
> "직관성 (Intuitiveness) - 최우선: 복잡한 에이전트 활동을 한눈에 파악"
> "상태 인식 속도: 3초 이내에 현재 상태 파악"

**실제 코드 현실:**
```javascript
// renderer.js 278-313
const pokeMessages = [
  "앗, 깜짝이야!",
  "열심히 일하는 중입니다!",
  "코드 짜는 중... 💻",
  "커피가 필요해요 ☕",
  // ...
];
```

**비판:** 이게 "직관성"인가? "업무 생산성"인가?
- 캐릭터 클릭 시 "깜짝 놀라는 메시지"가 업무 생산성에 어떤 도움을 주는가?
- PRD는 "생산성 극대화"를 외치지만, 코드는 "귀여운 반응"을 구현
- **이것은 사용자 문제를 해결하는 도구가 아니라, 개발자의 재미를 위한 장난감**

**문제 2: 접근성 재앙 (법적 리스크)**

**PRD 주장:**
> "접근성 긴급 개선 (P0)"
> "폰트 크기 하한선 10px"

**실제 코드 현실:**
```css
/* styles.css 229-248 */
.type-tag {
  font-size: 8.5px;  /* PRD 위반 */
  font-weight: 800;
  padding: 1px 5px;
}

/* styles.css 420-440 */
.agent-name {
  font-size: 10px;    /* 간신히 통과 */
  font-weight: 700;
}

/* styles.css 602-616 */
.agent-sub-badge {
  font-size: 7px;     /* PRD 심각 위반 */
  font-weight: 800;
}
```

**비판:** PRD는 "이번 주 내 수정"을 주장하지만, 코드는 7px 폰트를 여전히 사용
- **WCAG AA 위반**: 7px 폰트는 시각 장애인에게 읽기 불가능
- **법적 리스크**: 미국 장애인법(ADA), 유럽 접근성법 위반 가능성
- **PRD 신뢰도 붕괴**: "긴급 개선"이라고 쓰고 실제로는 수정 안 함

**문제 3: 기술 부채 무시 (PRD vs 현실)**

**PRD 주장:**
> "메모리 누수 수정 (긴급)"
> "각 에이전트별 개별 interval → 통합 애니메이션 루프"

**실제 코드 현실:**
```javascript
// renderer.js 82-106: 각 에이전트가 개별 interval 생성
const interval = setInterval(() => {
  // 애니메이션 로직
}, 1000 / sequence.fps);

// renderer.js 141-148: 각 에이전트가 개별 타이머 interval 생성
agentState.timerInterval = setInterval(() => {
  const elapsed = Date.now() - agentState.startTime;
  agentState.lastFormattedTime = formatTime(elapsed);
  bubble.textContent = `${config.label} (${agentState.lastFormattedTime})`;
}, 1000);
```

**비판:** 10개 에이전트 × 2개 interval = 20개 동시 실행
- **메모리 누수**: `removeAgent()`로 정리하지만, race condition 가능성
- **CPU 낭비**: 20개 interval이 1초마다 DOM 조작
- PRD는 "requestAnimationFrame 통합"을 주장하지만, 여전히 setInterval 사용

**문제 4: 사용자 설정 UI 부재**

**PRD 주장:**
> "사용자 설정 UI (단순 버전) (예상 16시간)"
> "테마 선택 (다크/라이트), 윈도우 위치, 캐릭터 크기"

**실제 코드 현실:**
```javascript
// renderer.js 549-595: 유일한 "설정"은 웹 대시보드 버튼
function createWebDashboardButton() {
  const button = document.createElement('button');
  button.id = 'web-dashboard-btn';
  button.innerHTML = '🌐 View as Web';
}
```

**비판:** 사용자는 아무것도 커스터마이즈할 수 없다
- 테마? 없음 (어두운 배경만 강제)
- 윈도우 위치? 없음 (하단 고정)
- 캐릭터 크기? 없음 (90px 고정)
- **"사용자 커스터마이징"이라는 PRD 가치 제안은 거짓말**

**문제 5: 정보 아키텍처 붕괴**

**PRD 주장:**
> "복잡한 에이전트 활동을 한눈에 파악"

**실제 코드 현실:**
```javascript
// renderer.js 399-519: updateGridLayout()
// - 500줄짜리 복잡한 레이아웃 로직
// - 중첩 if-else, 변수명: lastProject, mainIndex, fallbackSubList
```

**비판:** "한눈에 파악"이라는 거짓말
- **정보 과부하**: 프로젝트명, 타입 배지, 상태 말풍선, 타이머, 캐릭터, 이름 뱃지
- **시각적 잡음**: 10개 에이전트가 6개 정보 레이어를 동시에 표시
- **인지 부하**: 사용자는 3초 안에 상태를 파악하지 못함 (PRD 목표 실패)

**문제 6: 에이전트 넘버링 모순**

**실제 코드 현실:**

**비판:** 이게 사용자에게 의미 있는 정보인가?
- "Main_0", "Main_1"은 무엇을 의미하는가?
- 프로젝트별 구분이 아니라, 단순 순서 번호
- **사용자는 "내 프로젝트 A"를 찾고 싶은데, "Main_3"을 보고 있다**

## 2. 토론 쟁점

### 쟁점 1: "이 제품은 생산성 도구인가, 장난감인가?"

**내 주장:** 이 제품은 장난감이다. 생산성 도구가 아니다.

**근거:**

1. **비즈니스 가치 부족**
   - 캐릭터 클릭 시 "앗, 깜짝이야!" 메시지가 업무 효율을 높이는가?
   - 진짜 생산성 도구는: Slack의 "Status", Notion의 "Time Tracking", Jira의 "Burndown Chart"
   - 이 제품은: "귀여운 캐릭터가 춤추는 화면"

2. **사용자 니즈 미스매치**
   - PRD 페르소나 "김실험"은 "전체 현황 한눈에 보고 싶음"
   - 실제 구현: 10개 캐릭터가 6개 정보 레이어로 난장판
   - 김실험은 이걸 보고 "와, 생산성 올랐다"고 말할까?

3. **PMF 부재 증거**
   - PRD는 "2주안에 10명 인터뷰"를 주장
   - 하지만 코드에는 사용자 트래킹, 피드백 수집 기능이 전혀 없음
   - **"이 도구가 필요한가?"라는 질문을 던질 기회 자체를 만들지 않음**

**다른 전문가에게 던질 질문:**
> "귀하의 팀원이 10개 터미널을 열고 작업 중일 때, 귀하는 '앗, 깜짝이야!' 메시지를 보고 생산성이 올랐다고 느끼시겠습니까? 아니면 웃기다고 생각하시겠습니까?"

**예상 반론:** "하지만 귀여운 UI가 업무 스트레스를 줄여줍니다!"

**내 대응:** 그렇다면 그건 "장난감"이지 "생산성 도구"가 아닙니다. PRD는 "생산성과 즐거움을 극대화"라고 쓰지만, 실제로는 "즐거움"만 있고 "생산성"은 없습니다. 솔직하게 "업무 스트레스를 줄이는 장난감"이라고 포지셔닝하세요. "생산성 도구"라고 속이지 마세요.

---

### 쟁점 2: "PRD는 실행 가능한 계획인가, 판타지 소설인가?"

**내 주장:** 이 PRD는 판타지 소설이다. 실행 가능한 계획이 아니다.

**근거:**

1. **모순적인 우선순위**
   - PRD: "접근성 긴급 개선 (이번 주)"
   - 현실: 폰트 7px 여전히 존재
   - **"긴급"이라는 단어의 의미를 모르는 것인가? 아니면 그냥 써본 것인가?**

2. **현실 불가능한 일정**
   - PRD: "1개월 내 테스트 커버리지 20% → 3개월 50% → 6개월 60%"
   - 현실: 현재 0% 커버리지
   - **테스트 가능한 아키텍처로의 리팩토링 없이 어떻게 테스트를 쓰나?**
   - renderer.js 689줄 전역 코드를 어떻게 테스트하나?

3. **기능 우선순위의 혼란**
   - PRD: Mission Control은 "보류" (기술 부채 해결 후)
   - 현실: renderer.js 549-595에 Mission Control 버튼 이미 구현
   - **"보류"라는 말의 의미를 모르는 것인가?**

**다른 전문가에게 던질 질문:**
> "PRD는 '이번 주 내 폰트 10px 수정'을 주장합니다. 하지만 코드는 여전히 7px을 사용합니다. 이 PRD를 쓴 사람은 정말 수정할 의도가 있었던 것인가요, 아니면 그냥 '좋은 말'만 늘어놓은 것인가요?"

**예상 반론:** "PRD는 비전을 제시하는 문서입니다. 실행은 개발팀의 몫입니다."

**내 대응:** 비전과 "거짓말"은 다릅니다. "이번 주 내 수정"이라고 쓰고 3개월 동안 안 수정하면, 그건 "비전"이 아니라 "거짓말"입니다. PRD는 팀을 위한 "약속"입니다. 약속을 지킬 생각이 없다면, 아예 쓰지 마세요.

---

### 쟁점 3: "이 제품의 타겟 사용자는 실제로 존재하는가?"

**내 주장:** 타겟 사용자는 존재하지 않는다. PRD의 상상 속 페르소나일 뿐이다.

**근거:**

1. **CLI 순수주의자는 거부할 것**
   - PRD: "CLI 순수주의자는 제외"
   - 하지만 Claude CLI 사용자 = 대부분 CLI 순수주의자
   - **진짜 CLI 파워 유저는 tmux를 쓰지, 귀여운 캐릭터를 안 본다**

2. **AI 얼리 어답터는 필요 없다고 할 것**
   - PRD: "AI 도구 얼리 어답터 (20-35세)"
   - 하지만 AI 얼리 어답터는 "자동화"를 원하지, "시각화"를 원하지 않음
   - **"이 에이전트가 뭐 하는지 보려면 캐릭터 클릭하세요"라는 건, 자동화가 아니라 수동 모니터링입니다**

3. **팀 리드/매니저는 쓸모없다고 할 것**
   - PRD: "2차 타겟: 팀 리드/매니저"
   - 하지만 팀 리드는 Jira, Linear, Notion을 쓰지, 귀여운 캐릭터 화면을 안 봄
   - **Mission Control이 "보류"된 상태에서, 팀 리드가 이걸 왜 씁니까?**

**다른 전문가에게 던질 질문:**
> "PRD는 'AI 얼리 어답터'를 타겟으로 잡았습니다. 하지만 실제 Claude CLI 사용자 중 몇 %가 '귀여운 픽셀 아트 캐릭터'를 보고 싶어 할까요? 1%? 0.1%? 아니면 0%인가요?"

**예상 반론:** "청해 시장(Blue Ocean)입니다. 경쟁자가 없는 건 기회입니다!"

**내 대응:** 청해 시장이 아니라, **아무도 안 가는 사막**입니다. 경쟁자가 없는 건 "기회"가 아니라 "수요가 없다는 증거"입니다. tmux는 20년 된 도구입니다. 그동안 아무도 "터미널 시각화"를 안 만든 건, 아무도 원치 않아서입니다.

## 4. 결론

### 내 비판적 요약

1. **이 제품은 생산성 도구가 아니라 장난감이다**
   - 캐릭터 클릭 시 "깜짝 놀라는 메시지"가 업무 효율을 높이지 않는다
   - 사용자 니즈와 제품 기능 사이에 근본적인 불일치가 있다

2. **PRD는 실행 가능한 계획이 아니라 판타지 소설이다**
   - "이번 주 내 수정"이라고 쓰고 3개월 동안 안 수정한다
   - 약속을 지킬 생각이 없다면, 아예 쓰지 마라

3. **타겟 사용자는 존재하지 않는다**
   - CLI 순수주의자는 거부할 것이다
   - AI 얼리 어답터는 필요 없다고 할 것이다
   - 팀 리드는 쓸모없다고 할 것이다
